# PS01 — Intelligent Document Processing & Upload Optimization Engine
**IIT Roorkee Blockathon · Design Spec · 2026-03-14**

---

## 1. Problem Summary

Government portals like Apuni Sarkar (Uttarakhand) require citizens to upload identity documents (Aadhaar, PAN, Income Certificate, Domicile Certificate, Caste Certificate). Two bottlenecks exist:

1. **Oversized uploads** — citizens upload 5 MB+ files causing slow uploads on limited mobile networks and storage bloat.
2. **Manual verification** — officers visually inspect each document, a slow and error-prone process.

---

## 2. Solution Overview

A monolith FastAPI pipeline that:
1. **Compresses** uploaded documents transparently (client sees no change in flow)
2. **Runs OCR** via Google Vision API (Hindi + English mixed script)
3. **Extracts structured fields** per document type
4. **Cross-validates** fields across documents
5. **Writes a tamper-proof verification record** to a Polygon smart contract
6. **Returns** a JSON response with compressed files, extracted fields, confidence scores, and blockchain receipt

A React frontend provides a citizen upload screen and an officer dashboard.

---

## 3. Architecture

### 3.1 System Flow

```
Citizen → React Upload UI → POST /api/verify (multipart) → FastAPI Server
                                                                  │
                                                          ① Compress (Pillow / PyMuPDF)
                                                                  │
                                                          ② OCR (Google Vision API)
                                                                  │
                                                          ③ Extract Fields (regex + rules)
                                                                  │
                                                          ④ Cross-Validate (name/DOB/district)
                                                                  │
                                                          ⑤ Blockchain (web3.py → Polygon)
                                                                  │
Officer ← React Dashboard ← JSON + tx_hash ←────────────────────┘
```

### 3.2 Design Pattern

**Monolith pipeline** — single FastAPI service, synchronous per-request processing. Chosen for hackathon speed and simplicity. Each pipeline stage is an isolated Python module with a clean interface, making it straightforward to extract into microservices later.

## 4. Pipeline Stages

### Stage 1 — Compression (`compress.py`)

| Input | Output |
|---|---|
| Raw uploaded bytes (JPEG/PNG/PDF) | Compressed bytes + `{original_size, compressed_size, format}` |

- **JPEG/PNG**: Pillow — resize to max 1200px on longest edge, quality=70, progressive encoding
- **PDF**: PyMuPDF — render each page at 150 DPI, recompress as JPEG, reassemble PDF
- **Target**: < 300 KB per document while preserving text and QR code legibility
- **Format detection**: by MIME type, fallback to file magic bytes
- **Guarantee**: never increase file size (skip compression if already small)

### Stage 2 — OCR (`ocr.py`)

| Input | Output |
|---|---|
| Compressed image/PDF bytes | Full text string + per-word `{text, confidence, bounding_box}` |

- Google Cloud Vision `DOCUMENT_TEXT_DETECTION` endpoint
- Multi-language hint: `["hi", "en"]` for Hindi + English
- PDF handling: convert each page to PNG (PyMuPDF), OCR page by page, concatenate
- Returns raw full text + structured word annotations for downstream field extraction

### Stage 3 — Field Extraction (`extract.py`)

| Input | Output |
|---|---|
| OCR text + document type hint | `{doc_type, fields: {field_name: {value, confidence}}}` |

**Document type classifier**: keyword matching on OCR text
- "आधार" / "UIDAI" / "Unique Identification" → `aadhaar`
- "Income Tax" / "Permanent Account" / "PAN" → `pan`
- "Income Certificate" / "आय प्रमाण" → `income_cert`
- "Domicile" / "मूल निवास" → `domicile_cert`
- "Caste" / "जाति प्रमाण" → `caste_cert`

**Per-doc field extractors (regex)**:

| Document | Extracted Fields |
|---|---|
| Aadhaar | name, dob, gender, address, district (parsed from address), uid (last 4 digits) |
| PAN | name, dob, pan_number, father_name |
| Income Certificate | name, annual_income, district, issuing_authority |
| Domicile Certificate | name, address, district, issuing_authority |
| Caste Certificate | name, caste, sub_caste, district |

Confidence per field = average OCR word confidence for tokens matching that field.

### Stage 4 — Cross-Validation (`validate.py`)

| Input | Output |
|---|---|
| List of `DocumentResult` | `{checks: [{field, status, documents, values}], overall_confidence}` |

**Validation rules**:
- `name_match` — fuzzy string match (Levenshtein ≥ 85%) across all docs that have a name field
- `dob_match` — exact match after normalising date formats, across Aadhaar + PAN
- `district_match` — case-insensitive match across Domicile + Income Certificate

**Status values**: `MATCH` | `MISMATCH` | `MISSING`

**Overall confidence score** (0–100), where both inputs are in range `[0.0, 1.0]`:
```
score = ((weighted_avg_ocr_confidence × 0.5) + (cross_val_pass_rate × 0.5)) × 100
```

### Stage 5 — Blockchain (`blockchain.py`)

| Input | Output |
|---|---|
| applicant_id, doc_hashes[], overall_confidence, is_verified | `{tx_hash, block_number, contract_address}` |

**Smart contract** (Solidity, deployed on Polygon Amoy testnet):
```solidity
struct VerificationRecord {
    string  applicantId;
    bytes32[] docHashes;      // SHA-256 of each compressed document
    uint8   confidence;       // 0–100
    bool    isVerified;       // confidence >= 75
    uint256 timestamp;
}
mapping(string => VerificationRecord[]) public records;

function storeVerification(
    string memory applicantId,
    bytes32[] memory docHashes,
    uint8 confidence,
    bool isVerified
) external;

function getRecords(string memory applicantId)
    external view returns (VerificationRecord[] memory);
```

- Deploy with **Hardhat** to Polygon Amoy testnet (Mumbai was deprecated in 2024)
- Interact from Python with **web3.py**
- `is_verified = confidence >= 75`
- Polygonscan Amoy link returned in response for officer to view on-chain proof

## 5. API

### `POST /api/verify`

**Request**: `multipart/form-data`
- `applicant_id` (string)
- `files[]` (1–5 files, JPEG/PNG/PDF, max 20 MB each)

**Response** `200 OK`:
```json
{
  "applicant_id": "UK-2024-00183",
  "documents": [
    {
      "doc_type": "aadhaar",
      "original_size": 5042312,
      "compressed_size": 287440,
      "fields": {
        "name":    { "value": "Ramesh Kumar Negi", "confidence": 0.98 },
        "dob":     { "value": "14/03/1988",        "confidence": 0.96 },
        "uid":     { "value": "XXXX XXXX 4821",    "confidence": 0.99 },
        "district":{ "value": "Chamoli",           "confidence": 0.94 }
      }
    }
  ],
  "cross_validation": [
    { "field": "name_match",     "status": "MATCH",    "documents": ["aadhaar","pan"], "values": {"aadhaar":"Ramesh Kumar Negi","pan":"Ramesh K Negi"} },
    { "field": "dob_match",      "status": "MATCH",    "documents": ["aadhaar","pan"], "values": {"aadhaar":"14/03/1988","pan":"14-03-1988"} },
    { "field": "district_match", "status": "MATCH",    "documents": ["domicile_cert","income_cert"], "values": {"domicile_cert":"Chamoli","income_cert":"Chamoli"} }
  ],
  "overall_confidence": 91,
  "is_verified": true,
  "blockchain": {
    "tx_hash": "0x3f8a...d94c",
    "block_number": 47823910,
    "contract_address": "0xABCD...1234",
    "polygonscan_url": "https://amoy.polygonscan.com/tx/0x3f8a...d94c"
  }
}
```

### `GET /api/verifications`
Returns list of all past verifications (officer dashboard list view).
Requires header: `X-Officer-Key: <hardcoded demo key>` (demo-grade auth, not production).

**Response** `200 OK`:
```json
{
  "verifications": [
    {
      "applicant_id": "UK-2024-00183",
      "overall_confidence": 91,
      "is_verified": true,
      "tx_hash": "0x3f8a...d94c",
      "created_at": "2026-03-14T10:23:00Z"
    }
  ]
}
```

### `GET /api/verifications/{applicant_id}`
Returns full verification detail for a single applicant (same shape as `POST /api/verify` response).
Requires header: `X-Officer-Key: <hardcoded demo key>`.

## 6. Data Storage

**SQLite** (single file, `db/verifications.db`) — sufficient for hackathon demo.

Tables:
- `verifications(id, applicant_id, overall_confidence, is_verified, tx_hash, created_at)`
- `documents(id, verification_id, doc_type, original_size, compressed_size, fields_json)`
- `cross_validation(id, verification_id, field, status, documents_json, values_json)`

No separate DB server needed.

## 7. Frontend

### Citizen Upload Screen
- Application ID text input
- Full name text input
- Drag-and-drop upload zone (accepts JPEG/PNG/PDF, multiple files)
- Document type chips showing upload status (pending / ready)
- Submit button with shimmer animation
- On submit: loading state → redirect to result page

- Summary stat cards: Pending / Verified / Flagged counts
- Scrollable application list with status badges (VERIFIED / REVIEW / FLAGGED)

## 11. Error Handling

| Scenario | Behaviour |
|---|---|
| Unsupported file format | 400 with `"unsupported_format"` error |
| File > 20 MB | 413 with `"file_too_large"` error |
| Google Vision API failure | 502, pipeline aborts, returns partial results if any |
| Doc type unrecognised | Returns `doc_type: "unknown"`, skips field extraction, still compresses |
| Blockchain tx failure | Returns result without blockchain fields, logs error; verification still saved to SQLite |
| All docs missing a field | Cross-validation check marked `MISSING`, not `MISMATCH` |

## 12. Success Criteria (Demo)

- [ ] 5 MB Aadhaar JPEG compresses to < 300 KB without visible text degradation
- [ ] Extracted fields (name, DOB, UID) correct for a real Aadhaar scan
- [ ] Name mismatch between two docs correctly flagged as MISMATCH
- [ ] Verification record visible on Polygonscan Amoy after submission
- [ ] End-to-end latency < 10 seconds for a 3-document set
- [ ] Officer dashboard reflects new verifications after manual refresh (no polling required for demo)
