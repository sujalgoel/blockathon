Architecture Approaches

Pick the overall system design for PS01. Each has different trade-offs for hackathon speed vs. completeness.

A

Monolith Pipeline Recommended

Single FastAPI service handles compress → OCR → validate → blockchain in one request. React frontend talks to one endpoint.

React UI
  ↓ POST /verify (multipart files)
FastAPI Server
  ├── compress.py (Pillow/PyMuPDF)
  ├── ocr.py (Google Vision API)
  ├── validate.py (cross-field rules)
  └── blockchain.py (web3.py → Polygon)
  ↓ returns JSON + tx hash
Officer Dashboard

Pros

  • Fastest to build
  • Easy to debug
  • Single deploy

Cons

  • Blocking — no async stages
  • Hard to scale later
B

Microservices

Separate services for compression, OCR, validation, and blockchain. Communicate via REST or message queue.

React UI
  ↓
API Gateway
  ├── compress-service (port 8001)
  ├── ocr-service (port 8002)
  ├── validation-service (port 8003)
  └── blockchain-service (port 8004)
Docker Compose

Pros

  • Clean separation
  • Scalable per stage

Cons

  • Overkill for hackathon
  • More infra to wire up
  • Debugging across services
C

Async Queue Pipeline

FastAPI accepts upload, queues job in Redis/Celery, worker processes async. WebSocket pushes status to UI.

React UI ←── WebSocket (live status)
  ↓ POST /upload
FastAPI → Redis Queue
            ↓
Celery Worker
  compress → ocr → validate → chain
  ↓ result stored in DB

Pros

  • Non-blocking UI
  • Live progress updates
  • Realistic architecture

Cons

  • Redis + Celery overhead
  • More moving parts to demo
